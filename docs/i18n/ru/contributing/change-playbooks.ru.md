# Сценарии изменений

Пошаговые руководства по типовым расширениям и доработкам ZeroClaw.

Полные примеры кода для каждого расширяемого трейта см. в [extension-examples.md](../../../contributing/extension-examples.md).

## Добавление провайдера

- Реализуйте `Provider` в `src/providers/`.
- Зарегистрируйте фабрику в `src/providers/mod.rs`.
- Добавьте узкие тесты на подключение фабрики и ошибки.
- Не допускайте утечки поведения, специфичного для провайдера, в общий оркестратор.

## Добавление канала

- Реализуйте `Channel` в `src/channels/`.
- Сохраняйте единообразие `send`, `listen`, `health_check` и семантики индикатора набора.
- Покройте тестами авторизацию, allowlist и health.

## Добавление инструмента

- Реализуйте `Tool` в `src/tools/` со строгой JSON-схемой параметров.
- Проверяйте и санитизируйте все входы.
- Возвращайте структурированный `ToolResult`; избегайте panic на горячем пути.

## Добавление периферии

- Реализуйте `Peripheral` в `src/peripherals/`.
- Периферия предоставляет `tools()` — каждый инструмент делегирует железу (GPIO, датчики и т.д.).
- При необходимости зарегистрируйте тип платы в схеме конфигурации.
- Протокол и прошивка: `docs/hardware/hardware-peripherals-design.md`.

## Изменения безопасности / рантайма / шлюза

- Укажите угрозы/риски и стратегию отката.
- Добавьте или обновите тесты и доказательства для сбоев и границ.
- Наблюдаемость должна быть полезной и не содержать секретов.
- Для `.github/workflows/**` опишите влияние на allowlist Actions и обновите `docs/contributing/actions-source-policy.md`, если меняются источники.

## Документация / README / IA

- Навигация по документации — часть UX: README → хаб документации → SUMMARY → индекс разделов.
- Верхний уровень навигации держите коротким; избегайте дублирующих ссылок в соседних блоках.
- При изменении поверхности рантайма обновите ссылки в `docs/reference/`.
- При смене навигации или ключевых формулировок синхронизируйте входные точки для всех локалей (`en`, `zh-CN`, `ja`, `ru`, `fr`, `vi`).
- Общие правки текста — в том же PR с локализациями (или явно зафиксируйте отложение и follow-up PR).

## Общее состояние инструментов

- Для долгоживущего состояния используйте паттерн `Arc<RwLock<T>>`.
- Хендлы передавайте при конструировании; не вводите глобальное/статическое изменяемое состояние.
- Для изоляции по клиенту используйте `ClientId` от демона — не собирайте ключи идентичности внутри инструмента.
- Чувствительные данные (учётные записи, квоты) — строго по клиенту; широковещательное состояние можно делить с опциональным префиксом пространства имён.
- После смены конфигурации кэш валидации сбрасывается — инструмент обязан повторно проверить данные перед следующим запуском после сигнала.
- Полный контракт: [ADR-004: Tool Shared State Ownership](../../../architecture/adr-004-tool-shared-state-ownership.md).

## Цикл инструментов агента, QueryEngine и хуки

- **Единый путь инструментов:** `run_tool_call_loop` в `src/agent/loop_.rs` всегда заходит в `run_query_loop` в `src/agent/query_engine.rs`, который пишет диагностику [`TransitionReason`](../../../../src/agent/state.rs) и по успеху запускает пост-ходовые хуки **`void` + `blocking`** (`src/agent/stop_hooks.rs`). Cargo-фичи `query_engine_v2` **нет**; путь всегда включён.
- **Компакция:** обрезка истории перед вызовом LLM — `src/agent/compaction_pipeline.rs` (именованные стадии + `history_pruner`); после обрезки может готовиться Markdown-фрагмент **memory reload** (дайджест сессии + фрагмент индекса AutoMemory) для динамического хвоста; реактивные повторы контекста там же, где это подключено из цикла.
- **Системный промпт:** каноническая сборка в `src/agent/system_prompt.rs` (мемоизированный статический префикс + изменяемый хвост; маркер разбиения `__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__`). `build_system_prompt_*` в `src/channels/mod.rs` делегирует сюда; часть путей передаёт `system_prompt_refresh` в `run_tool_call_loop`, чтобы `src/agent/loop_.rs` обновлял `history[0]` после `run_pre_llm_phases`. `src/providers/anthropic.rs` мапит маркер в два system-блока для кэширования промпта. Внутрипроцессная статистика: `crate::agent::query_engine::last_system_prompt_assembly` и `zeroclaw doctor query-engine` (при `[memory.layered]` также выводит статистику селектора слоистой памяти).
- **Сначала транскрипт:** пользовательские строки для JSONL сессии фиксируйте через `session_transcript::commit_user_turn` на границе оркестрации до работы модели (каналы и `Agent::turn` / `turn_streamed` следуют этому).
- **Построение HookRunner:** `crate::hooks::hook_runner_from_config` (`src/hooks/mod.rs`) регистрирует встроенные хуки при `[hooks].enabled` и по-прежнему регистрирует **`MemoryConsolidationHook`**, если `memory.auto_save` истинно (даже при выключенных хуках), чтобы имя хука в конфигах оставалось стабильным — обработчик **пустой**, **consolidation выполняется с `await`** на основном пути QueryEngine / Agent (`query_engine.rs`, `agent.rs`), без второго вызова LLM. Запись SessionMemory / AutoMemory при `[memory.layered]` и запущенной consolidation — через `src/memory/consolidation.rs`; слот ожидающего хода — `src/memory/layered_context.rs`.
- **Шлюз:** в `run_gateway` создайте `HookRunner`, сохраните в `AppState.hooks`, передайте `state.hooks.clone()` в `Agent::from_config_with_hooks` для `/ws/chat`, чтобы пост-ходовые хуки совпадали с каналами.
- **Потоковый sink хода:** `run_tool_call_loop` / `run_query_loop` принимают опциональный `turn_event_sink` (`Sender<TurnEventSink>`): [`TurnEventSink::DeltaText`](../../../../src/agent/agent.rs) — черновики/прогресс из цикла инструментов; [`TurnEventSink::Emit`](../../../../src/agent/agent.rs) оборачивает [`TurnEvent`](../../../../src/agent/agent.rs) для чанков модели и телеметрии инструментов. [`Agent::turn_streamed`](../../../../src/agent/agent.rs) использует тот же тип; [`src/gateway/ws.rs`](../../../../src/gateway/ws.rs) мапит в WebSocket JSON (`chunk`, `tool_call`, `tool_result`, затем `chunk_reset` и `done`). Протокол: [`.claude/skills/zeroclaw/references/rest-api.md`](../../../../.claude/skills/zeroclaw/references/rest-api.md).
- **Расширение пост-ходового поведения:** реализуйте `HookHandler::on_after_turn_completed` / `after_turn_completed_blocking` (аргументы `user_message` и `assistant_summary`); регистрируйте на том же `HookRunner`, что и шлюз или каналы.

## Правила архитектурных границ

- Расширяйте через новые реализации трейтов + фабрики; избегайте переписывания модулей ради одной фичи.
- Зависимости направлены внутрь к контрактам: конкретика зависит от трейтов/конфига/утилит, а не от другой конкретики.
- Избегайте сквозной связности (провайдер тянет внутренности канала, инструмент меняет политику шлюза напрямую).
- Одна ответственность на модуль: оркестрация — `agent/`, транспорт — `channels/`, I/O модели — `providers/`, политика — `security/`, исполнение — `tools/`.
- Новые общие абстракции — после правила трёх повторов и хотя бы одного реального вызова.
- Изменения схемы/конфига — публичный контракт: дефолты, совместимость, миграция и откат.
